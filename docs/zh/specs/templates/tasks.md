## 模板指南

### 任务结构要求

每个任务必须包含：
- **明确的目标**: 需要编写或修改哪些具体代码
- **实现细节**: 要处理的具体文件、组件或模块
- **需求可追溯性**: 引用具体需求（例如，_需求: 1.2, 3.1_）
- **可操作范围**: 任务应可由编码代理完成

### 任务层级

最多使用两个层级：
- **顶级任务**: 主要实现阶段或史诗 (Epics)
- **子任务**: 具体的编码活动（用小数点编号：1.1, 1.2, 2.1）

### 可选任务

使用“*”后缀标记可选任务：
- **测试任务**: 基于属性的测试、单元测试、集成测试
- **文档任务**: API 文档、代码注释（非核心功能时）
- **增强任务**: 非核心需求的加分项功能

**重要提示**: 只有子任务可以标记为可选，顶级任务绝不可以。

### 基于属性的测试任务

对于设计文档中的每个正确性属性：
- 创建一个单独的子任务
- 包含来自设计的确切属性描述
- 引用正在验证的需求
- 将属性测试放在相关实现任务附近

格式：
```
- [ ]* X.Y 为 [功能] 编写属性测试
  - **属性 [N]: [来自设计的确切属性文本]**
  - **验证需求: [X.Y]**
```

### 检查点任务

在逻辑中断处包含检查点任务：
- 在核心功能实现后
- 在主要集成点之前
- 在实现结束时

格式：
```
- [ ] X. 检查点 - [简要描述]
  - 确保所有测试通过，如有疑问请询问用户。
```

### 任务排序

组织任务以：
- **增量构建**: 每个任务都建立在之前的工作之上
- **尽早验证**: 在相关实现附近包含测试任务
- **最小化依赖**: 减少任务之间的阻塞
- **支持反馈**: 包含用于验证的检查点

### 需求可追溯性

每个任务必须引用具体需求：
- 使用格式：`_需求: [X.Y, X.Z]_`
- 引用细粒度的验收标准，而不仅仅是用户故事
- 确保所有需求都至少由一个任务覆盖
- 在逻辑任务集群中对相关需求进行分组

### 仅关注编码

仅包含涉及以下内容的任务：
- 编写新代码
- 修改现有代码
- 编写测试代码
- 设置开发基础设施

**排除**如下任务：
- 用户验收测试 (UAT)
- 生产部署
- 性能指标收集
- 业务流程变更
- 营销活动

### 实现优先方法

遵循此顺序：
1. 实现核心功能
2. 编写相应的测试
3. 验证并迭代
4. 进入下一个功能

这确保了在进行全面测试之前已经存在可运行的代码。

### 任务完成标准

每个任务应该是：
- **具体的**: 明确要编写什么代码
- **可衡量的**: 任务完成时显而易见
- **可实现的**: 可由编码代理完成
- **相关的**: 直接支持需求
- **有时限的**: 范围限定在合理的实现工作量内

### 与规格说明系统的集成

任务应与以下内容保持一致：
- **需求**: 每个任务都追溯到具体的验收标准
- **设计**: 实现遵循记录的架构
- **属性**: 基于属性的测试验证设计正确性属性
- **质量门控**: 检查点确保符合质量标准
