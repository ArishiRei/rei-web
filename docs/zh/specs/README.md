# 规格驱动开发

本目录包含项目的规格驱动开发系统。规格驱动开发是一种基于正式规格说明书进行功能开发的方法论，包含需求、设计和实现计划。

## 概述

规格系统帮助团队：
- 通过清晰的需求系统性地规划功能
- 通过适当的架构考虑设计解决方案
- 通过结构化任务分解实现功能
- 维护需求和实现之间的可追溯性
- 通过正式验证流程确保质量

## 规格工作流程

### 1. 需求阶段
使用 `templates/requirements.md` 中的模板创建需求文档。此文档应包括：
- 清晰的介绍和功能概述
- 术语和系统定义词汇表
- 带有验收标准的用户故事
- 遵循 EARS（Easy Approach to Requirements Syntax）模式的需求

### 2. 设计阶段
使用 `templates/design.md` 中的模板创建设计文档。此文档应包括：
- 系统架构和组件设计
- 数据模型和接口
- 用于验证的正确性属性
- 错误处理策略
- 测试方法

### 3. 实现阶段
使用 `templates/tasks.md` 中的模板创建任务文档。此文档应包括：
- 带有清晰目标的编号任务分解
- 每个任务的需求可追溯性
- 适用的基于属性的测试任务
- 用于验证的检查点任务

## 入门指南

1. **复制模板**: 从 `templates/` 复制适当的模板到您的功能目录开始
2. **参考示例**: 参考 `examples/` 中的示例获取结构和内容指导
3. **维护可追溯性**: 确保每个设计元素都能追溯到需求，每个任务都引用特定需求
4. **持续验证**: 使用每个模板中的验证标准确保质量

## 模板使用

### 需求模板
需求模板遵循 EARS 模式和 INCOSE 质量规则：
- 每个需求必须遵循六种 EARS 模式之一
- 所有技术术语必须在词汇表中定义
- 需求必须是主动语态、可测量且不涉及解决方案

### 设计模板
设计模板包括：
- 用于基于属性测试的正确性属性
- 组件接口和数据模型
- 架构决策和理由
- 结合单元测试和基于属性测试的测试策略

### 任务模板
任务模板提供：
- 带有清晰目标的结构化任务分解
- 需求可追溯性部分
- 用 "*" 标记的可选测试任务
- 用于验证的检查点任务

## 质量指南

- **完整性**: 所有模板必须完全填写
- **可追溯性**: 每个设计元素都应追溯到需求
- **可测试性**: 包括单元测试和基于属性的测试
- **清晰性**: 始终使用清晰、可操作的语言
- **一致性**: 遵循既定的模式和术语

## 与开发流程的集成

规格系统与以下内容集成：
- **协议文档**: 与 `docs/protocol/` 中的开发标准保持一致
- **架构指南**: 遵循 `docs/architecture/` 中定义的模式
- **质量门禁**: 支持协议文档中定义的验证流程
- **构建系统**: 规格可以作为构建流程的一部分进行验证

有关开发协议和标准的更多信息，请参阅 [协议文档](../protocol/README.md)。
